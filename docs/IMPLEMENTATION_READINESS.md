# PL-timeTabler 구현 준비도 결정서

## 결론

상태는 **구현 시작 가능(READY)** 이다.

- 제품 범위, 핵심 사용자 흐름, 서비스 경계, 데이터 원칙, 모바일 UI 원칙은 확정했다.
- React + TypeScript, FastAPI, PostgreSQL, Docker Compose, 별도 optimizer라는 핵심 기술 스택은 확정했다.
- OR-Tools CP-SAT은 서버 자동 생성기의 **주 solver로 채택**한다.
- 단, solver 선택은 이름만으로 확정하지 않고 실제 대진대 fixture로 성능·정확성·재현성을 통과해야 운영 확정으로 간주한다.
- 구현을 막는 제품 기획 공백은 없다. 세부 라이브러리 버전과 운영 서비스 선택은 해당 단계에서 검증·고정한다.

따라서 모든 세부 결정을 더 작성할 때까지 기다리지 않고, 기반 구축과 첫 세로 기능 구현을 시작한다.

## 1. 확정된 결정

### 제품

- 대진대학교 전용
- 로그인 없이 수동 시간표 편집부터 시작
- 모바일 우선, 데스크톱에서도 같은 기능 제공
- 자동 생성은 사용자를 대신해 결정하는 기능이 아니라 조건을 반영한 후보 비교 기능
- 졸업요건은 공식 공개 근거가 있는 범위만 판정하고 불확실한 규칙은 `미확인` 처리
- 실제 수강신청 실행, 학교 계정 수집, 실시간 잔여석, 강의평은 범위에서 제외

### 기술

| 영역 | 결정 |
|---|---|
| 프론트엔드 | React + TypeScript 기반 SPA |
| API | Python + FastAPI |
| 자동 생성 | 별도 Python optimizer + OR-Tools CP-SAT |
| 데이터베이스 | PostgreSQL 18 |
| DB 계층 | SQLAlchemy 2 + Alembic |
| 계약 | OpenAPI 3.1 snapshot + 생성 TypeScript SDK |
| 배포 | 역할별 컨테이너를 Docker Compose로 통합 |
| 즉시 계산 | 브라우저 TypeScript 충돌 검사·대체 분반 탐색 |
| 무거운 계산 | 서버 optimizer의 전체 후보 생성 |

프론트, API, optimizer, DB를 한 이미지나 한 프로세스에 합치지 않는다. API와 optimizer는 같은 backend 코드와 이미지를 공유할 수 있지만 실행 프로세스와 장애 경계는 분리한다.

## 2. OR-Tools 결정

### 채택 이유

시간표는 단순 정렬보다 다음 조건을 동시에 다루는 이산 제약 최적화 문제다.

- 시간 충돌, 중복 과목, 잠금, 필수과목, 최소·최대 학점 같은 hard constraint
- 공강일, 빈 시간, 오전·야간 회피, 이동 부담, 기존 선택 보존 같은 soft constraint
- 서로 비슷하지 않은 복수 후보
- 제한 시간 내 최선의 실행 가능 해 반환

OR-Tools CP-SAT은 이 조건을 한 모델 안에서 명시적으로 표현하기에 적합하고 Python backend와도 바로 결합된다.

### 혼합 방식

OR-Tools만으로 모든 상호작용을 처리하지 않는다.

1. **브라우저 계산:** 추가 즉시 충돌 표시, 가능한 다른 분반, 작은 국소 교체
2. **서버 CP-SAT:** 여러 필수·선호·졸업요건을 반영한 전체 후보 생성
3. **공통 fixture:** 양쪽의 시간 구간과 충돌 판정을 교차 검증

이 구조는 수동 편집의 즉시성과 자동 생성의 표현력을 모두 확보한다.

### 대안 판단

| 대안 | 판단 |
|---|---|
| 자체 DFS/backtracking만 사용 | 작은 문제의 기준 구현에는 유용하지만 제약과 설명·다양성 요구가 늘수록 유지가 어렵다. 테스트용 기준 solver로만 사용한다. |
| SciPy/HiGHS MILP | 선형 모델에는 충분히 강하지만 시간표의 조건부 제약과 다양한 후보 표현은 CP-SAT이 더 직접적이다. CP-SAT 실패 시 재검토한다. |
| Timefold Solver | 학교 시간표 사례가 있고 강력하지만 Java 21 런타임과 세 번째 backend 언어가 추가된다. 현재 구조에는 운영 복잡성 대비 이점이 작다. |
| LLM 기반 생성 | hard constraint의 정확성과 재현성을 보장하지 못하므로 solver로 사용하지 않는다. 문구 설명에도 초기에는 필요하지 않다. |

### 운영 확정 게이트

Phase D 전에 실제 데이터로 다음을 통과한다.

- 정상·경계·불가능 조건을 포함한 최소 50개 optimizer fixture
- 충돌·잠금·필수과목·학점 hard constraint 위반 0건
- 대표 입력 자동 생성 p95 3초 이하, hard timeout 8초
- 동일 입력·데이터 버전·solver 버전·seed에서 후보와 순서 재현
- 최소 3개 후보가 실질적으로 다른 시간표 형태를 가짐
- 시간 제한 시 `해 없음`과 `시간 안에 못 찾음`을 구분
- 실패 시 완화 가능한 조건을 사용자 문장으로 반환
- 작은 fixture에서 기준 backtracking 결과와 실행 가능성 교차 검증

게이트를 통과하지 못하면 먼저 후보 분반 축소, 대칭 제거, 목적함수 단계화, 검색 파라미터를 개선한다. 그 뒤에도 목표를 만족하지 못할 때만 MILP나 제한 범위 전용 탐색으로 교체를 검토한다.

## 3. 실용 기능 우선순위

### P0 — 첫 출시 필수

1. **즉시 검색·분반 비교**: 과목명, 교수, 코드, 학과, 교양영역 검색과 같은 과목 분반 묶음 비교
2. **안전한 수동 편집**: 추가, 삭제, 분반 교체, 잠금, 다단계 실행 취소
3. **충돌 즉시 해결**: 충돌 원인과 가능한 대체 분반을 같은 화면에 표시
4. **과목 의도 구분**: `반드시`, `희망`, `예비`, `제외`로 후보의 역할 지정
5. **현실적인 선호조건**: 목표 학점, 원하는 공강일, 오전·야간 회피, 하루 최대 수업량, 최소 점심시간
6. **비교 가능한 자동 후보**: 3개 이상의 후보와 등교일, 총 공강시간, 첫·마지막 수업, 변경 과목, 미충족 선호 표시
7. **수강신청 실패 대비안**: 1순위 분반을 놓쳤을 때 미리 지정한 예비 분반·과목으로 한 번에 교체
8. **졸업요건 연결**: 부족한 필수·교양영역·전공영역에 해당하는 이번 학기 개설과목 추천과 공식 근거 표시
9. **데이터 신뢰 표시**: 학기, 갱신일, 출처, 시간·강의실 미정, 이전 저장 이후 변경된 분반 경고
10. **로컬 자동 저장**: 로그인 없이 브라우저에 저장하고 오프라인·느린 네트워크에서도 마지막 정상 상태 사용
11. **수강신청 준비 내보내기**: 과목코드·분반·과목명을 순서 있는 체크리스트로 복사하고 시간표 이미지·인쇄 제공
12. **불가능 조건 완화 안내**: 어떤 조건 때문에 해가 없는지와 가장 작은 완화 선택지를 제시

### P1 — 출시 직후 가치가 큰 기능

- 시간표 후보 나란히 비교
- 건물 간 이동시간과 연속 수업 이동 위험
- 폐지·대체·재수강 과목 경고
- 만료·삭제 가능한 공유 링크와 복사 편집
- PWA 설치와 더 강한 오프라인 캐시
- 캘린더 내보내기

### P2 — 데이터·개인정보 조건이 갖춰진 뒤

- 친구와 공통 공강 계산
- 안전한 이수내역 파일 가져오기
- 개인 선호 가중치의 반복 사용 최적화

### 현재 제외

- 채팅형 AI 도우미와 근거 없는 `AI 추천 점수`
- 신뢰할 수 없는 좌석·강의평 데이터
- 학교 포털 로그인·자동 수강신청
- 소셜 피드, 배지, 불필요한 게임화

## 4. 남은 결정과 처리 시점

핵심 아키텍처의 미확정이 아니라 구현 시 검증해 lockfile과 ADR에 고정할 세부사항이다.

### 첫 기반 작업에서 고정

- OR-Tools wheel과 호환되는 Python base image 정확한 버전·digest
- React 빌드 도구, 패키지 관리자, 스타일링 방식, 접근성 primitive의 최소 구성
- Python·TypeScript lint, typecheck, unit, E2E 도구와 CI 명령
- PostgreSQL job 점유, heartbeat, timeout, retry의 트랜잭션 규칙
- generated SDK 갱신과 breaking-change 검사 명령

권장 기본값은 Vite 기반 SPA, CSS 변수와 범위가 제한된 CSS, 검증된 접근성 primitive의 선택 사용이다. 큰 UI kit와 과도한 스타일 추상화는 사용하지 않는다.

### 해당 기능 직전에 고정해도 되는 사항

- 이미지와 PDF 중 첫 내보내기 우선순위
- 공유 링크 만료 기본값
- 강의실 좌표·건물 간 이동시간 데이터 방식
- 이수내역 파일 형식과 개인정보 처리 정책
- 배포 호스팅, 도메인, 컨테이너 registry, 외부 모니터링 공급자

이 항목들은 현재 로컬 구현 시작을 막지 않는다.

## 5. 구현 시작 순서

### 0 — 짧은 기술 검증

버리는 화면을 만드는 단계가 아니라 최종 구조에서 위험도가 높은 부분을 먼저 잠근다.

1. 모노레포, lockfile, Compose, CI 기본 골격
2. 실제 강의 데이터의 시간 문자열 → `Session[]` 파서와 fixture
3. 브라우저 충돌 판정의 공통 fixture
4. 작은 기준 backtracking과 CP-SAT의 모델·성능 비교
5. 카탈로그와 optimizer 요청·응답 OpenAPI 계약

### 첫 번째 세로 기능

`2026 카탈로그 로드 → 모바일 검색 → 분반 추가 → 시간표 표시 → 충돌 경고 → 로컬 저장`

이 흐름을 먼저 완료하면 데이터, API, 생성 SDK, 디자인 시스템, 모바일 상호작용, 테스트와 Docker 경계를 한 번에 검증할 수 있다.

### 이후 세로 기능

1. 분반 교체·잠금·대체 분반·실행 취소
2. 과목 의도와 선호조건 입력
3. CP-SAT 복수 후보 생성·비교·설명·완화
4. 졸업요건 부족 영역과 이번 학기 과목 연결
5. 예비 수강신청안, 공유·내보내기
6. 데이터 갱신 운영, 접근성·성능·복구·사용성 출시 검증
