# PL-timeTabler 제품 완성형 실행 계획

> 구현 시작 판단, OR-Tools 대안·검증 게이트, P0/P1/P2 기능 우선순위는 [`IMPLEMENTATION_READINESS.md`](IMPLEMENTATION_READINESS.md)를 기준으로 한다.

## 1. 제품 정의

### 확정 방향

- 대상: 대진대학교 학부생
- 기본 흐름: 선택형 학과 온보딩을 건너뛸 수 있는 모바일 우선 수동 편집, 로그인은 선택
- 자동화: 충돌 해결 보조와 조건 기반 전체 시간표 자동 생성
- 학사 기능: 입학연도·학과·전공 방식별 졸업요건과 부족 영역 안내
- 개발 기준: 일회성 MVP가 아니라 운영 가능한 최종 구조·디자인 시스템·데이터 이력·QA를 처음부터 적용

현재 2026학년도 1학기 과목 1,576개, 강의실 325개, 수업 세션 1,693개가 준비되어 있다. 과목·강의실 결합 가능 분반은 1,336개이며 결측은 정상 상태로 모델링한다.

### 제품 범위의 경계

완성형은 모든 학교 기능을 복제한다는 뜻이 아니다. 다음 범위 안에서 출시 품질을 달성한다.

- 포함: 학과·학년 온보딩, 공식 필수과목 연결, 선택형 학교 이메일 OTP 로그인, 강의 검색, 분반 비교, 충돌 검사, 수동 편집, 자동 생성, 공강·빈 시간·시간대·이동 선호, 졸업요건 안내, 저장·공유·내보내기, 데이터 버전·근거 표시
- 제외: 실제 수강신청 실행, 학교 계정 비밀번호 수집, 비공개 성적 자동 조회, 실시간 잔여석, 강의평, 타 대학 지원
- 원칙: 공식 공개 자료로 검증할 수 없는 졸업요건은 `미확인`으로 표시하고 졸업 가능 여부를 확정하지 않는다.

## 2. 핵심 사용자 흐름

### 바로 만드는 흐름

1. 첫 방문에는 학과·입학연도·학년을 설정하거나 `건너뛰고 바로 만들기`를 선택한다.
2. 프로필이 있으면 메인 편집기에서 입학연도·전형과 검증 자료가 일치하는 전공필수와 교양필수의 실제 개설 분반을 확인한다. 다른 cohort와 편입학은 임의 적용하지 않고 수동 확인으로 남긴다.
3. 필수과목은 자동으로 넣지 않고 사용자가 원하는 분반을 배치한다.
4. 과목명·교수명·과목코드·교양영역·학과로 나머지 과목을 검색한다.
5. 같은 과목의 분반을 시간·교수·강의실과 함께 비교한다.
6. 분반을 추가하면 충돌·총 학점·등교일·빈 시간을 즉시 갱신한다.
7. 충돌 시 원인과 가능한 대체 분반을 한 화면에서 해결한다.

### 자동 생성 흐름

1. 사용자는 반드시 들을 과목, 후보 과목, 희망 학점과 선호조건을 입력한다.
2. 잠근 과목·피할 시간·원하는 공강일을 지정한다.
3. 서버 optimizer가 실행 가능 후보를 만들고 진행 상태를 보여준다.
4. 후보별 등교일, 빈 시간, 첫·마지막 수업, 변경 내용, 미충족 선호를 비교한다.
5. 후보 전체 또는 일부를 적용한 뒤 수동으로 수정하고 언제든 되돌린다.

### 졸업요건 흐름

1. 입학연도, 학과·전공, 단일·복수·부전공 방식을 선택한다.
2. 총학점, 교양필수·영역, 전공기초·필수·선택, 논문·시험 요건을 확인한다.
3. 기존 이수과목은 브라우저에서 직접 선택하거나 파일 가져오기가 안전하게 정의된 뒤 선택적으로 입력한다.
4. 부족한 영역에서 이번 학기 개설과목을 바로 후보에 추가한다.
5. 각 규칙에서 공식 문서, 적용 연도, 페이지·조항, 갱신일을 확인한다.

### 저장·공유·내보내기

- 편집 상태는 버전이 있는 로컬 스키마로 자동 저장한다.
- 공유 링크에는 학기·분반·잠금·표시 조건만 포함하고 개인 이수내역은 넣지 않는다.
- 공유안을 복사해 자신의 시간표로 편집할 수 있다.
- 이미지와 인쇄 친화 PDF 내보내기를 제공하며 학기와 데이터 기준일을 표시한다.

## 3. 기술 결정

- 프론트엔드: React + TypeScript
- API: Python + FastAPI
- 최적화: 별도 Python optimizer + OR-Tools CP-SAT
- 데이터베이스: PostgreSQL 18
- DB 계층: SQLAlchemy 2 + Alembic migration
- 계약: FastAPI OpenAPI 3.1에서 TypeScript SDK 생성
- 실행: `web`, `api`, `optimizer`, `db`, `migrate`, `ingest`를 Docker Compose로 구성
- 상세 경계와 서버 이동 계약: `docs/ARCHITECTURE.md`

API와 optimizer는 같은 backend 이미지와 도메인 모델을 공유하지만 별도 프로세스로 실행한다. Python API의 I/O 처리와 CPU-bound solver가 서로를 막지 않게 하는 필수 경계다.

## 4. 도메인과 최적화 계약

### 핵심 상태

```text
Course            courseCode, name, category, credits
Section           courseCode, sectionCode, professor, sessions[], room?
CurriculumVersion admissionYearRange, academicUnit, majorMode, sources[]
RequirementRule   kind, min/maxCredits, courseSet, groupLogic, exceptions
DraftTimetable    selectedSections[], lockedSections[], preferences, history[]
OptimizationJob   inputSnapshot, status, deadline, candidates[], explanation
```

- 내부 분반키: `${curiNo}-${clssNo}`
- 강의실 미상과 시간 미정은 오류가 아니라 명시적 nullable 상태다.
- 학과 명칭 변경·통폐합, 폐지·대체과목, 입학연도별 규칙을 유효기간으로 보존한다.
- 모든 졸업요건 판정은 공식 `sourceDocument`로 역추적할 수 있어야 한다.

### Hard constraints

- 수업시간 충돌 금지, 동일 과목 중복 금지
- 잠긴 분반과 필수 지정 과목 유지
- 희망 최소·최대 학점
- 검증된 교양·전공 필수 그룹과 선수과목
- 학년·학과·캠퍼스 제한은 원문이 있을 때만 적용

### Soft constraints

- 등교일과 수업 사이 빈 시간 최소화
- 원하는 공강일, 오전·야간 회피, 선호 교수·과목
- 연속 수업 이동 부담과 하루 수업 편중 최소화
- 기존 선택 변경 최소화
- 상위 후보가 같은 형태로 수렴하지 않도록 다양성 확보

같은 입력·규칙 버전·solver 버전·seed는 같은 후보 순서를 반환해야 한다. 결과 상태는 `OPTIMAL`, `FEASIBLE`, `INFEASIBLE`, `TIME_LIMIT`, `FAILED`를 구분하고 불가능한 이유를 가능한 범위에서 설명한다.

## 5. 구현 단계

각 단계는 최종 구조의 일부이며 버리는 프로토타입을 만들지 않는다. 다만 선행 검증이 끝나기 전에는 후속 기능을 임의로 확장하지 않는다.

### A — 제품 기반·디자인 시스템·CI·Docker

작업:

1. React와 Python backend 모노레포, lockfile, lint·format·typecheck·test를 구성한다.
2. Compose로 web/API/optimizer/PostgreSQL/migration/ingest를 연결한다.
3. FastAPI OpenAPI snapshot과 TypeScript SDK 생성을 CI에 포함한다.
4. 색상·타이포·간격·상태·모션 토큰과 핵심 컴포넌트 상태를 정의한다.
5. 구조화 로그, request/job ID, healthcheck, 오류 경계를 만든다.
6. dependency·container 취약점 검사와 secret scan을 적용한다.

완료 기준:

- 새 checkout에서 문서화된 한 명령으로 개발 스택 기동
- lint, typecheck, unit, build, migration, contract test 성공
- 360/390/768/1440px 앱 셸에 불필요한 가로 스크롤 없음
- API 장애와 optimizer 장애가 서로의 프로세스를 중단시키지 않음

### B — 강의·졸업요건 데이터 파이프라인

작업:

1. 개설과목과 강의실 데이터를 `(curiNo, clssNo)`로 결합한다.
2. 복수 요일·시간 문자열을 분 단위 `Session[]`으로 정규화한다.
3. 공식 학칙·대학생활안내·교육과정편람·학과 문서의 source inventory를 만든다.
4. 입학연도·학과·전공방식별 규칙과 전공필수·교양영역·졸업평가를 정규화한다.
5. raw 원문, checksum, 게시·시행일, parser version, 검수 상태를 보존한다.
6. import를 staging에서 검증한 뒤 트랜잭션으로 활성 버전을 전환한다.
7. 변경 diff와 누락·충돌을 검수하는 관리자용 report를 생성한다.

완료 기준:

- 과목 1,576개, 강의실 325개, 세션 1,693개 수량 보존
- 모든 레코드가 성공·시간 미정·검수 필요 중 하나로 분류
- 같은 입력 재수집 시 checksum이 같고 중복 생성 없음
- 졸업요건마다 적용 대상과 공식 원문 링크 존재
- 확인되지 않은 규칙은 자동으로 충족 판정에 사용되지 않음

### C — 모바일 우선 검색·수동 편집기

작업:

1. 시간표, 검색 바텀시트, 선택 과목 목록을 단일 편집 흐름으로 만든다.
2. 이름·교수·코드·학과·이수구분·교양영역 검색과 필터를 제공한다.
3. 추가·삭제·분반 변경·잠금·다중 실행 취소를 구현한다.
4. 충돌 그래프와 대체 분반을 브라우저에서 즉시 계산한다.
5. 시각 격자 외에 스크린리더용 요일별 구조화 목록을 제공한다.
6. 느린 네트워크에서도 마지막 정상 카탈로그와 편집 상태를 사용할 수 있게 한다.

완료 기준:

- 1,576개 데이터 검색 p95 100ms 이하
- 검색 결과에서 두 번 이하 탭으로 과목 추가
- 360px에서 정밀 드래그 없이 모든 핵심 편집 가능
- 키보드로 검색·분반 선택·시트 닫기·삭제·실행 취소 가능
- 일반 텍스트와 상태 표현이 WCAG 2.2 AA 기준 충족

### D — OR-Tools 자동 생성·설명 가능한 추천

작업:

1. hard/soft constraint와 가중치를 도메인 규칙으로 분리한다.
2. PostgreSQL 기반 optimization job의 점유·heartbeat·취소·재시도를 구현한다.
3. CP-SAT 모델과 후보 다양성 제약, solver timeout, 결정론적 seed를 적용한다.
4. 수동 개선과 전체 자동 생성을 같은 후보 비교 UI에 연결한다.
5. 후보마다 목적함수의 구성값과 변경 이유를 사람이 이해할 문장으로 만든다.
6. 불가능·시간초과·일부 선호 미충족 상태를 구분한다.

완료 기준:

- 충돌·잠금·필수과목 hard constraint 위반 후보 0건
- 대표 후보군에서 자동 생성 p95 3초 이하, hard timeout 8초
- 같은 입력·버전·seed에서 후보와 순서 재현
- API 부하 중 solver 실행이 일반 조회 latency를 유의미하게 막지 않음
- 적용 후 한 번의 실행 취소로 이전 시간표 복원

### E — 졸업요건 안내와 과목 연결

작업:

1. 입학연도·학과·단일/복수/부전공 프로필을 만든다.
2. 총학점·교양·전공·졸업논문/시험 요건을 계층적으로 표시한다.
3. 기존 이수내역을 브라우저에서 검색·선택해 입력한다.
4. 부족한 규칙 그룹을 이번 학기 개설과목과 연결한다.
5. 폐지·대체과목과 학과 명칭 변경을 버전 규칙으로 처리한다.
6. 판정 옆에 공식 원문·페이지·적용연도·기준일을 표시한다.

완료 기준:

- 공식 fixture의 기대 충족·부족 결과와 100% 일치
- 학과·입학연도가 다른 규칙이 섞이지 않음
- 미확인·충돌 규칙은 경고와 문의 경로를 표시
- 개인 이수내역은 동의 없이 서버나 공유 링크로 전송되지 않음

### F — 저장·공유·내보내기

작업:

1. 버전이 있는 로컬 저장과 안전한 schema migration을 구현한다.
2. 데이터 학기 변경과 분반 삭제 시 적용 가능 항목만 부분 복원한다.
3. 추측하기 어려운 공유 ID, 만료·삭제 정책, rate limit을 적용한다.
4. 공유 시간표 복사, 이미지·인쇄 PDF 내보내기를 제공한다.
5. PostgreSQL backup·restore와 새 서버 이전 스크립트를 만든다.

완료 기준:

- 새로고침·브라우저 재시작 후 편집 상태 복원
- 손상된 저장값에도 앱이 안전하게 시작
- 공유 payload에 개인 이수내역·사용자 식별정보 없음
- 새 DB에 복원 후 카탈로그·규칙·공유·작업 수와 checksum 일치

### G — 운영 데이터 갱신과 관측성

작업:

1. 강의계획서·개설과목 공개 시 source fetch → diff → 검수 → 활성화 절차를 자동화한다.
2. parser 실패, 규칙 변경, 과목 미연결, 오래된 출처를 dashboard/report로 확인한다.
3. API latency·오류율·job queue·solve time·import 상태를 수집한다.
4. 백업 복원 훈련, rollback, 장애 runbook을 작성한다.
5. 학기 전 트래픽과 동시 최적화 작업을 재현하는 부하 테스트를 수행한다.

완료 기준:

- 새 학기 원문을 기존 활성 데이터 손상 없이 staging 검증
- 데이터 변경 내역을 source와 parser 버전까지 감사 가능
- 오류 알림에서 해당 import/job ID와 재현 입력 추적 가능
- 운영 secret·개인 데이터가 로그에 남지 않음

### H — 출시 품질 검증

작업:

1. 핵심 상태의 컴포넌트·시각 회귀와 모바일 E2E를 만든다.
2. 모바일 Safari, Android Chrome, 데스크톱 키보드 흐름을 검증한다.
3. 자동 접근성 검사와 수동 스크린리더 점검을 수행한다.
4. 대진대 학생을 입학연도·학과가 겹치지 않게 모집해 과제 테스트한다.
5. 추천 신뢰도, 졸업요건 이해도, 10초 이상 머뭇거림을 기록해 반복 개선한다.

완료 기준:

- serious/critical 접근성 자동 위반 0건
- 목표 뷰포트 핵심 E2E와 시각 회귀 통과
- 모바일 Lighthouse Performance·Accessibility 90점 이상
- 참여자 80% 이상이 도움 없이 3분 내 5과목 시간표 완성
- 참여자 80% 이상이 자동 후보의 차이와 졸업요건 근거를 올바르게 설명
- P0/P1 결함 0건, 데이터·복구·보안 출시 체크리스트 통과

## 7. 주요 위험과 대응

| 위험 | 대응 |
|---|---|
| 입학연도·학과별 규칙이 매우 다양함 | 규칙 DSL보다 먼저 source/version 모델을 고정하고 미확인은 확정 판정에서 제외 |
| 공식 문서가 PDF·표·이미지로 혼재 | raw 보존, OCR 결과와 원문 분리, 자동 파싱 뒤 사람 승인 |
| 학과 명칭 변경·통폐합 | `academic_units` 유효기간과 successor 관계로 보존 |
| solver가 느리거나 해를 못 찾음 | 후보군 사전 축소, timeout, feasible 우선 반환, job 분리와 부하 테스트 |
| 추천이 불투명해 신뢰 하락 | 점수 구성값·변경점·미충족 선호·공식 근거 공개 |
| Python CPU 작업이 API를 막음 | API와 optimizer 컨테이너·프로세스 완전 분리 |
| 모바일 정보 과밀 | 편집 우선, 세부 규칙·조건은 시트와 단계적 공개 |
| 개인 성적정보 노출 | 기본 브라우저 저장, 명시적 동의 전 서버 전송 금지, 공유 payload 분리 |

## 8. 실행 순서와 중단 조건

- 기본 순서: A → B → C → D → E → F → G → H
- B의 출처·버전 모델이 검증되기 전에는 졸업 가능 판정 UI를 만들지 않는다.
- 충돌 검사와 hard constraint fixture가 통과하기 전에는 선호 점수를 조정하지 않는다.
- API와 optimizer 분리 부하 테스트가 통과하기 전에는 solver를 동기 API로 합치지 않는다.
- 실제 학생 테스트에서 이해하지 못한 추천 설명은 기능 완료로 보지 않는다.
- H의 출시 기준과 복구 훈련을 모두 통과했을 때 제품 1.0 완료로 판단한다.
